Lab 4.4.8: Configure Secure Passwords and SSH

This lab focuses on implementing fundamental security measures including secure password policies, password encryption, and SSH configuration for encrypted remote management access. The configuration is divided between router RTA and switch SW1, with emphasis on replacing insecure Telnet with SSH.

Router RTA Configuration

Basic Setup & Interface Configuration

hostname RTA
interface GigabitEthernet0/0/0
 ip address 172.16.1.1 255.255.255.0
 no shutdown

Sets device hostname and configures IP addressing on interface

Password Security Configuration

service password-encryption
security passwords min-length 10
enable secret [secure_password]
no ip domain-lookup
ip domain-name netsec.com

Enables password encryption, minimum length requirement, and disables DNS lookup

SSH Configuration

username [username] secret [password]
crypto key generate rsa
! Specify 1024 bits when prompted
line vty 0 4
 transport input ssh
 login local
 exec-timeout 6
copy running-config startup-config

Creates user account, generates RSA keys, and configures VTY lines for SSH-only access

Switch SW1 Configuration

Basic Setup & VLAN Interface

hostname SW1
interface vlan 1
 ip address 172.16.1.2 255.255.255.0
 no shutdown

Configures switch hostname and management interface

Password Security

service password-encryption
security passwords min-length 10
enable secret [secure_password]
ip domain-name netsec.com

Same security policies as router configuration

SSH Configuration

username [username] secret [password]
crypto key generate rsa
! Specify 1024 bits when prompted
line vty 0 15
 transport input ssh
 login local
 exec-timeout 6 0
copy running-config startup-config

Switch supports more VTY lines (0-15) compared to router

Key Security Principles

Password Security

  • • Minimum 10-character password policy
  • • Encrypted password storage
  • • Strong enable secret protection

SSH Implementation

  • • RSA key-based encryption
  • • Telnet disabled (transport input ssh)
  • • Session timeout for idle connections

Lab 6.3.7: Configure OSPF Authentication

This lab implements OSPF MD5 authentication to secure routing updates between routers. The configuration ensures only authenticated routers can exchange routing information, preventing unauthorized route injection and enhancing network security.

Network Topology

graph LR R1["Router R1
G0/0/0"] -->|"Area 0"| R2["Router R2
G0/0/0"] R2["Router R2
G0/0/1"] -->|"Area 0"| R3["Router R3
G0/0/1"]

Router R1 Configuration 

conf t
interface g0/0/0
 ip ospf message-digest-key 1 md5 MD5pa55
router ospf 1
 area 0 authentication message-digest
end

Configures MD5 authentication key on interface and enables message-digest authentication for OSPF area 0

Router R2 Configuration 

conf t
interface g0/0/0
 ip ospf message-digest-key 1 md5 MD5pa55
interface g0/0/1
 ip ospf message-digest-key 1 md5 MD5pa55
router ospf 1
 area 0 authentication message-digest
end

Same configuration applied to both interfaces since R2 connects to both R1 and R3

Router R3 Configuration 

conf t
interface g0/0/1
 ip ospf message-digest-key 1 md5 MD5pa55
router ospf 1
 area 0 authentication message-digest
end

Single interface configuration as R3 only connects to R2

OSPF Authentication Key Points

Configuration Requirements

  • • Key ID must match on all routers (1 in this case)
  • • Password must be identical across all devices
  • • Applied per-interface in OSPF area

Security Benefits

  • • Prevents unauthorized route injection
  • • MD5 hashing prevents password transmission
  • • Maintains routing protocol integrity

Lab 7.4.9: Configure Server-based Authentication with TACACS+ and RADIUS

This lab implements centralized authentication using external AAA servers with TACACS+ (on R2) and RADIUS (on R3) protocols. Server-based AAA provides scalable authentication for large networks with centralized user management.

TACACS+ Configuration (Router R2)

Enable AAA and Configure Server

aaa new-model
tacacs-server host 192.168.2.2
tacacs-server key tacacspa55

Activates AAA and specifies TACACS+ server with shared secret

Authentication Method List

aaa authentication login default group tacacs+ local

Uses TACACS+ first, falls back to local database if server unavailable

Apply to VTY Lines

line vty 0 4
 login authentication default

Applies TACACS+ authentication to remote access lines

RADIUS Configuration (Router R3)

Enable AAA and Configure Server

aaa new-model
radius-server host 192.168.3.2
radius-server key radiuspa55

Activates AAA and specifies RADIUS server with shared secret

Authentication Method List

aaa authentication login default group radius local

Uses RADIUS first, falls back to local database if server unavailable

Apply to VTY Lines

line vty 0 4
 login authentication default

Applies RADIUS authentication to remote access lines

TACACS+ vs RADIUS Comparison

TACACS+ (Router R2)

  • • Cisco proprietary protocol
  • • Encrypts entire packet
  • • Separates authentication, authorization, accounting
  • • TCP port 49
  • • Better for device administration

RADIUS (Router R3)

  • • Open standard protocol
  • • Encrypts only password
  • • Combines authentication and authorization
  • • UDP ports 1812, 1813
  • • Better for network access

Lab 8.5.5: Configure Named Standard IPv4 ACLs

This lab focuses on configuring named standard IPv4 ACLs, which provide more flexibility and manageability than numbered ACLs. Named ACLs allow descriptive names and individual entry editing, making them ideal for production environments.

Router R2 Configuration

Deny Specific Network

ip access-list standard BLOCK_192.168.11.0
 deny 192.168.11.0 0.0.0.255

Creates named ACL and denies traffic from 192.168.11.0/24 network

Permit All Other Traffic

permit any

Explicit permit statement to allow all other traffic

Apply ACL to Interface

interface GigabitEthernet0/0
 ip access-group BLOCK_192.168.11.0 out

Applies ACL to interface in outbound direction

Router R3 Configuration

Deny Specific Network

ip access-list standard BLOCK_192.168.10.0
 deny 192.168.10.0 0.0.0.255

Creates named ACL and denies traffic from 192.168.10.0/24 network

Permit All Other Traffic

permit any

Explicit permit statement to allow all other traffic

Apply ACL to Interface

interface GigabitEthernet0/0
 ip access-group BLOCK_192.168.10.0 out

Applies ACL to interface in outbound direction

Named ACL Advantages

Manageability

  • • Descriptive names
  • • Easy identification
  • • Better documentation

Flexibility

  • • Edit individual entries
  • • No need to recreate entire ACL
  • • Insert/delete specific lines

Scalability

  • • No numeric limitations
  • • Better for large networks
  • • Easier troubleshooting

Lab 8.5.6: Configure Numbered Standard IPv4 ACLs

This lab covers the configuration of numbered standard IPv4 ACLs, the traditional method for implementing access control on Cisco devices. While less flexible than named ACLs, numbered ACLs remain important for understanding fundamental access control concepts.

Router R2 Configuration

Deny Network

access-list 1 deny 192.168.11.0 0.0.0.255

ACL 1 denies traffic from 192.168.11.0/24 network

Permit All

access-list 1 permit any

Allows all other traffic not explicitly denied

Apply ACL

interface GigabitEthernet0/0
 ip access-group 1 out

Applies ACL 1 to interface in outbound direction

Router R3 Configuration

Deny Network

access-list 1 deny 192.168.10.0 0.0.0.255

ACL 1 denies traffic from 192.168.10.0/24 network

Permit All

access-list 1 permit any

Allows all other traffic not explicitly denied

Apply ACL

interface GigabitEthernet0/0
 ip access-group 1 out

Applies ACL 1 to interface in outbound direction

Numbered ACL Limitations

Configuration Constraints

  • • Limited to numbers 1-99, 1300-1999
  • • Cannot edit individual entries
  • • Must delete and recreate entire ACL to modify
  • • No descriptive names for identification

Operational Considerations

  • • Less readable in large configurations
  • • Harder to troubleshoot
  • • Limited documentation value
  • • Being phased out in modern networks

Lab 8.5.12: Configure Extended ACLs - Scenario 1

This lab demonstrates the configuration of both numbered and named extended ACLs for granular traffic filtering. The scenario involves controlling access from two employee PCs (PC1 and PC2) to a central server with specific service restrictions.

Scenario Requirements

PC1 Access Requirements

  • • FTP access to server (TCP port 21)
  • • ICMP access for connectivity testing
  • • No communication with PC2

PC2 Access Requirements

  • • HTTP access to server (TCP port 80)
  • • ICMP access for connectivity testing
  • • No communication with PC1

Numbered Extended ACL Configuration (PC1)

Permit FTP Traffic

access-list 100 permit tcp 172.22.34.64 0.0.0.31 host 172.22.34.62 eq ftp

Allows FTP (port 21) from PC1's network to server

Permit ICMP Traffic

access-list 100 permit icmp 172.22.34.64 0.0.0.31 host 172.22.34.62

Allows ICMP (ping) from PC1's network to server

Apply ACL to Interface

interface gigabitEthernet 0/0
 ip access-group 100 in

Applies ACL inbound on interface connected to PC1's network

Named Extended ACL Configuration (PC2)

Create Named ACL

ip access-list extended HTTP_ONLY

Enters named ACL configuration mode

Permit HTTP Traffic

permit tcp 172.22.34.96 0.0.0.15 host 172.22.34.62 eq www

Allows HTTP (port 80) from PC2's network to server

Permit ICMP Traffic

permit icmp 172.22.34.96 0.0.0.15 host 172.22.34.62

Allows ICMP (ping) from PC2's network to server

Apply Named ACL

interface gigabitEthernet 0/1
 ip access-group HTTP_ONLY in

Applies ACL inbound on interface connected to PC2's network

Extended ACL Security Features

Granular Control

  • • Source and destination IP
  • • Protocol types (TCP, UDP, ICMP)
  • • Port numbers (FTP, HTTP, etc.)

Placement Strategy

  • • Close to source (extended ACLs)
  • • Inbound filtering preferred
  • • Conserves router resources

Application Benefits

  • • Service-specific restrictions
  • • Host-to-host control
  • • Network segmentation

Lab 8.5.13: Configure Extended ACLs - Scenario 2

This advanced lab presents a complex security scenario requiring granular control over multiple hosts and services. A single named extended ACL is configured on router RT1 to implement specific access restrictions for three PCs accessing two internet servers.

Security Policy Requirements

PC1 Restrictions

  • • Block HTTP (port 80)
  • • Block HTTPS (port 443)
  • • Both Server1 and Server2

PC2 Restrictions

  • • Block FTP (port 21)
  • • Both Server1 and Server2

PC3 Restrictions

  • • Block ICMP (ping)
  • • Both Server1 and Server2

Named Extended ACL Configuration

Deny PC1 HTTP/HTTPS Access (4 rules)

deny tcp host 172.31.1.101 host 64.101.255.254 eq www
deny tcp host 172.31.1.101 host 64.101.255.254 eq 443

Blocks HTTP and HTTPS to Server1

deny tcp host 172.31.1.101 host 64.103.255.254 eq www
deny tcp host 172.31.1.101 host 64.103.255.254 eq 443

Blocks HTTP and HTTPS to Server2

Deny PC2 FTP Access (2 rules)

deny tcp host 172.31.1.102 host 64.101.255.254 eq ftp

Blocks FTP to Server1

deny tcp host 172.31.1.102 host 64.103.255.254 eq ftp

Blocks FTP to Server2

Deny PC3 ICMP Access (2 rules)

deny icmp host 172.31.1.103 host 64.101.255.254

Blocks ICMP to Server1

deny icmp host 172.31.1.103 host 64.103.255.254

Blocks ICMP to Server2

Permit All Other IP Traffic

permit ip any any

Allows all traffic not explicitly denied by previous rules

Apply and Verify ACL

interface g0/0
 ip access-group ACL in

Applies ACL inbound on LAN interface

show access-lists

Verifies ACL operation and hit counts

Multi-Host ACL Configuration Principles

Rule Order Importance

  • • Specific deny statements first
  • • General permit statements last
  • • Implicit deny all at end
  • • First match wins evaluation

Host Specification

  • • Use 'host' keyword for single IP
  • • No wildcard mask needed for hosts
  • • Precise targeting of devices
  • • Avoids unintended matches

Lab 8.6.5: Configure IP ACLs to Mitigate Attacks

This lab focuses on using ACLs as a security tool to mitigate common network attacks. The configuration includes both standard and extended ACLs designed to block malicious traffic patterns and protect network infrastructure.

Common Attack Mitigation ACLs

Block RFC 1918 Private Addresses (Inbound)

access-list 101 deny ip 10.0.0.0 0.255.255.255 any
access-list 101 deny ip 172.16.0.0 0.15.255.255 any
access-list 101 deny ip 192.168.0.0 0.0.255.255 any
access-list 101 permit ip any any

Prevents private address spoofing from external networks

Restrict VTY Access

access-list 10 permit host 192.168.3.3
                line vty 0 4
                access-class 10 in

Restricts VTY access to management PC only (uses access-class, not access-group)

Block Reserved/Loopback Addresses

access-list 102 deny ip 127.0.0.0 0.255.255.255 any
access-list 102 deny ip 0.0.0.0 255.255.255.255 any
access-list 102 permit ip any any

Blocks loopback and 0.0.0.0 source addresses

Block Fragmented Packets

access-list 120 deny ip any any fragments
access-list 120 permit ip any any

Prevents fragmentation attacks and reconnaissance

Block Common Service Attacks

access-list 130 deny tcp any any eq 135
access-list 130 deny tcp any any eq 139
access-list 130 deny tcp any any eq 445
access-list 130 permit ip any any

Blocks Windows RPC, NetBIOS, and SMB ports

ICMP Attack Mitigation

Rate Limit ICMP Echo Requests

ip access-list extended ICMP_LIMIT
 permit icmp any any echo
 permit icmp any any echo-reply
 deny icmp any any

interface [interface_id]
 ip access-group ICMP_LIMIT in
 rate-limit input access-group ICMP_LIMIT 8000 1500 2000 conform-action transmit exceed-action drop

Limits ICMP echo requests to prevent ping floods while allowing normal operation

TCP SYN Attack Protection

TCP Intercept Configuration

ip tcp intercept list 140
access-list 140 permit tcp any host [protected_server_ip] eq 80
ip tcp intercept mode intercept
ip tcp intercept max-incomplete high 500
ip tcp intercept max-incomplete low 400
ip tcp intercept drop-mode random

Protects against TCP SYN floods by intercepting and validating connections

ACL Attack Mitigation Best Practices

Ingress Filtering

  • • Block RFC 1918 addresses from outside
  • • Block reserved and multicast addresses
  • • Implement BCP 38 (anti-spoofing)
  • • Filter unnecessary services

DoS/DDoS Protection

  • • Rate limit ICMP and SYN packets
  • • Block known attack vectors
  • • Use TCP intercept features
  • • Monitor ACL hit counts

Lab 8.7.4: Configure IPv6 ACLs

This lab covers the configuration of IPv6 access control lists, which are essential for securing modern networks. IPv6 ACLs share similar concepts with IPv4 but have different syntax and address formats.

Basic IPv6 ACL Configuration

Create IPv6 ACL

ipv6 access-list BLOCK_IPV6_PRIVACY
 deny ipv6 2001:db8:acad::/48 any
 permit ipv6 any any

Creates IPv6 ACL to block specific IPv6 prefix while allowing all other traffic

Apply IPv6 ACL to Interface

interface [interface_id]
 ipv6 traffic-filter BLOCK_IPV6_PRIVACY in

Applies IPv6 ACL inbound on specified interface

IPv6 Extended ACL Configuration

Service-Specific Filtering

ipv6 access-list IPV6_SERVICES
 permit tcp 2001:db8:acad:10::/64 host 2001:db8:acad:30::10 eq www
 permit tcp 2001:db8:acad:10::/64 host 2001:db8:acad:30::10 eq 443
 permit icmp 2001:db8:acad:10::/64 host 2001:db8:acad:30::10
 deny ipv6 any any

Allows HTTP, HTTPS, and ICMP while denying all other IPv6 traffic

IPv6 Neighbor Discovery Protection

ipv6 access-list ND_PROTECTION
 permit icmp any any nd-na
 permit icmp any any nd-ns
 deny ipv6 any any

Allows Neighbor Discovery while blocking other IPv6 traffic

IPv6 ACL Verification Commands

Show IPv6 ACLs

show ipv6 access-list

Displays all configured IPv6 ACLs and their hit counts

Interface IPv6 ACLs

show ipv6 interface [interface_id]

Shows IPv6 ACLs applied to specific interface

IPv6 Traffic Statistics

show ipv6 traffic

Displays IPv6 traffic statistics including ACL drops

IPv6 ACL Key Differences from IPv4

Address Format

  • • IPv6 addresses in hexadecimal
  • • Prefix notation (/64, /48 etc.)
  • • No wildcard masks - use prefix lengths
  • • 'any' keyword same as IPv4

Protocol Considerations

  • • ICMPv6 essential for IPv6 operation
  • • Neighbor Discovery (ND) protocol
  • • Stateless Address Autoconfiguration
  • • IPv6 extension headers

Class Maps for Traffic Classification

Create Class Map
